# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A deployable Databricks AI Runtime (AIR) pipeline (originally a DAIS demo) for fine-tuning an open-source LLM with Unsloth LoRA, registering the merged model to Unity Catalog as a custom LLM, deploying it to Mosaic AI Model Serving (vLLM), and load testing the endpoint at high QPS. The worked example fine-tunes `unsloth/Qwen3-4B-Instruct-2507` on IBM TabFormer credit-card fraud transactions; customers swap in their own dataset, model, and workspace via config. The deliverables are Databricks notebooks plus YAML configs — there is no test suite or build step, but `scripts/validate_config.py` enforces the cross-file config contracts (run locally and in CI via `.github/workflows/validate.yml`).

## Commands

```bash
# Local environment (only needed for Databricks Connect execution)
python -m venv .venv && source .venv/bin/activate
pip install -r 01_train/requirements.txt

# Verify CLI auth (profile DEFAULT; workspace host is in databricks.yml)
databricks auth profiles

# Run the ingestion and training-data staging notebooks locally via
# Databricks Connect (serverless); 01 writes the transaction table, 02 builds
# the SFT table and stages the Parquet shard export training reads
.venv/bin/python 00_setup/01_load_tabformer_dataset.py
.venv/bin/python 00_setup/02_stage_training_data.py

# Submit training via the AI Runtime CLI (alternative to the notebook's @distributed cell).
# COPYFILE_DISABLE=1 is required on macOS: without it bsdtar synthesizes AppleDouble
# (._*) entries in the code snapshot tarball from file xattrs; the remote launcher
# resolves $CODE_SOURCE_PATH from the tarball's first entry and picks `._01_train`,
# failing with "can't open file '/databricks/code_source/._01_train/train.py'".
# The CLI's own --exclude='._*' can't catch these (they're synthesized at archive
# time, not read from disk), and bsdtar hides them from `tar -tzf` listings —
# inspect snapshots with Python tarfile.getnames() instead.
cd 01_train && COPYFILE_DISABLE=1 air run --file train.yaml --watch

# Validate the cross-file config contracts (no workspace connection needed)
python scripts/validate_config.py

# Bundle operations (Declarative Automation Bundle, target: dev).
# Set the workspace host in databricks.yml first — it ships with a placeholder.
databricks bundle validate
```

The training runner notebook (`01_train/runner.py`) **cannot run locally** — it requires Databricks Serverless GPU with the AI Runtime `AI v5` environment (`serverless_gpu.distributed`, H100/A10 GPUs). Run it as a workspace notebook. The load-test notebook runs on Databricks serverless compute.

## Databricks Notebook Source Format

Every `.py` file in `00_setup/`, `01_train/`, and `02_deploy/` is a Databricks notebook **except `01_train/train.py`**, which is a plain Python module holding the training implementation (imported by the training notebook and executed directly by the AI Runtime CLI per `01_train/train.yaml`). When editing notebooks, preserve the format: `# Databricks notebook source` header, `# COMMAND ----------` cell separators, `# MAGIC %md` markdown cells.

- Every consumer (the notebooks, `train.py`, local scripts) gets the shared helpers (`load_yaml_config`, `config_str`/`config_int`/`config_float`/`config_bool`, `quote_identifier`, `full_name`, `get_spark_session`, `load_training_config`, `init_training_workspace`) the same way: insert the `01_train/` directory into `sys.path`, then `from training_utils import ...`. Never `%run` it, and never add the `# Databricks notebook source` header to it — notebook-formatted files cannot be imported in the workspace (`NotebookImportException: Importing notebooks directly is not supported`), and `train.py` must be able to import the module under the runner notebook, the AIR CLI, and local runs alike.
- The helpers module must **never** be renamed back to `utils.py`: the Databricks AI GPU base environment's `nvidia_cutlass_dsl` package registers a top-level `utils` module once torch/CUDA loads (e.g. when `train.py` imports unsloth), poisoning `sys.modules` so a local `from utils import ...` resolves to cutlass's vendored module and fails with `ImportError: cannot import name ... from 'utils' (...nvidia_cutlass_dsl/...)`.
- Training config flows through `training_utils.load_training_config()`, which returns a flat dict that the runner notebook and `train.py` each bind with `globals().update(...)` — so both see identical constants (`MODEL_NAME`, `TRAINING_OUTPUT_DIR`, `sft_table_q`, ...). It reads `parameters.training_config` from the YAML file at `$HYPERPARAMETERS_PATH` when AIR sets it (so `air run --override` values reach the script), falling back to `train.yaml` next to `training_utils.py`. It is deliberately a function, not top-level module code, so importing `training_utils` has no side effects (no config parse, no Spark, no schema creation). `load_yaml_config` resolves paths against the module's own directory by default; callers in other folders (the load-test notebook) pass `base_dir=Path.cwd()` for their own YAML.
- Notebooks support dual execution: inside Databricks (serverless notebooks, where `Path.cwd()` is the notebook's folder) or locally via Databricks Connect. `get_spark_session()` in `training_utils.py` attaches a serverless Connect session when none exists; the setup notebooks resolve their own directory via `__file__` with a `dbutils` notebook-context fallback before importing the helpers from `01_train/`.

## Architecture: Five-Stage Pipeline Coupled by YAML Configs

Each stage is a notebook driven by a sibling YAML file, connected to the next stage only through Unity Catalog objects:

1. **Ingest & stage** — two notebooks share `00_setup/setup.yaml`, which is self-contained (the values it shares with `train.yaml` are validated for agreement by `scripts/validate_config.py`). `00_setup/01_load_tabformer_dataset.py` downloads TabFormer into a UC staging volume and writes the cleaned transaction table (`fraud_dataset`) with the prompt-ready `*_text`/`amount_usd`/`fraud_label` fields. `00_setup/02_stage_training_data.py` then builds the SFT table (`fraud_sft_dataset`) from it — `prompt`, `assistant_response`, and `shard_id` columns generated with Spark expressions — and exports the SFT records to the `training_data` volume as Parquet partitioned by `shard_id` (the AIR data-loading pattern for large Delta tables); it fails fast with guidance when the transaction table is missing, and verifies the export by reading it back through Spark (shard count and row count) so the check also works on local Connect runs. **Both overwrite their tables/export on every run.** Each creates the schema and its volumes if missing. A third setup notebook, `00_setup/03_download_base_model_weights.py`, snapshots Hugging Face models into UC volumes. Its download list is the `models:` section of `setup.yaml` — each entry pairs a Hugging Face repo id (`huggingface_path`) with the volume directory that receives the weights (`volume_path`) — so arbitrary open-source models can be mirrored, not just the training base model; a final advisory cell warns (without failing) when `train.yaml`'s `model_volume_path` is not covered by any entry. It must run as a workspace notebook (needs the `/Volumes` FUSE mount), and it stages each download on local disk before copying sequentially into the volume because volume FUSE mounts reject the random-offset writes `hf_transfer`/resumed downloads use. Training loads base weights from `model_volume_path` when it is non-empty (`MODEL_LOAD_PATH` in `load_training_config()`), falling back to downloading `model_name` from Hugging Face when it is empty; reruns skip a download when its volume snapshot is complete.
2. **Train** — `01_train/runner.py` + `01_train/train.yaml` (one file: top-level AIR CLI workload schema with the training configuration nested under `parameters.training_config`, the schema's designated field for custom structured config — the AIR CLI rejects unknown top-level keys): reads the SFT table, fine-tunes with Unsloth LoRA, saves rank-0 adapters to a UC volume, and logs the adapter location on the MLflow run as the `adapter_output_dir` parameter — the handle the deploy stage resolves. The training implementation is `01_train/train.py` (`run_rank_training()` trains one rank's shard slice; rank/world size are resolved launcher-agnostically); the notebook's `@distributed` cell (sized by `notebook_gpus`/`notebook_gpu_type`) is a thin wrapper that inserts `NOTEBOOK_DIR` into `sys.path` on each worker and calls it, and the AI Runtime CLI runs the same file standalone (`air run --file train.yaml`). Creates the schema and the checkpoint volume if they don't exist (the SFT table itself still comes from setup).
3. **Register / deploy** — `02_deploy/01_register_and_deploy.py` + `02_deploy/deploy.yaml`: selects a training run (`run_id`, or when empty the best FINISHED run in `experiment_name` by `best_run_metric`/`best_run_metric_goal`, considering only runs that logged both the metric and `adapter_output_dir`), merges the adapter into the base model with `load_unsloth_model` (imported from `01_train/train.py`), registers the merged model to UC as a custom LLM, and creates/updates the serving endpoint. Runs on Serverless GPU + AI v5 — the merge loads the base model.
4. **Load test** — `02_deploy/load_test/load_test_serving_endpoint.py` + `02_deploy/load_test/serving_load_test.yaml`: samples prompts from the SFT table, smoke-tests the endpoint, then generates paced async HTTP traffic from Spark tasks via `mapInPandas` + `aiohttp`. **Appends** a summary row to the results Delta table.
5. **Monitor** — `03_monitor/01_unpack_inference_table.py` + `03_monitor/monitor.yaml`: incrementally unpacks the endpoint's AI Gateway inference table (raw JSON `request`/`response` strings plus delivery metadata) into an analysis-ready Delta table via Structured Streaming with `trigger(availableNow=True)` and a checkpoint in a UC volume — each run processes only new payload rows, so the notebook is schedulable. It parses the OpenAI chat payloads (prompt = last user message, completion, finish reason, token usage), optionally extracts structured-output fields into columns (`response_json_fields`), and creates the unpacked table with change data feed enabled + partitioned by `request_date` (what data profiling wants for incremental refreshes). Deployment always enables inference logging (`put_ai_gateway` runs inside the deploy function, so an endpoint is never deployed without its payload table; AI Gateway inference tables are the recommended capture path for custom model endpoints — the legacy `auto_capture_config` is retired and once AI Gateway tables are enabled an endpoint cannot switch back; delivery is best-effort within ~1 hour, and 4xx/5xx requests may not be logged). A second module (data-profiling monitor creation over the unpacked table) is planned.

### Cross-file contracts (keep these in sync)

- **Config coupling**: each stage's YAML is self-contained so the modules run standalone; the shared values are duplicated by design and `scripts/validate_config.py` enforces every agreement: `catalog`/`schema` across all five YAMLs (training values under `train.yaml`'s `parameters.training_config`); `setup.yaml`'s `table` = `train.yaml`'s `source_table`; `sft_table` across setup/train/load test; `sft_volume` between `setup.yaml` and `train.yaml` (it locates the Parquet export training reads); `endpoint_name` between `deploy.yaml` and `serving_load_test.yaml` (deploy creates the endpoint the load test targets); `deploy.yaml`'s `experiment_name` = `train.yaml`'s (run selection searches the experiment training logs to; absolute workspace paths are exempt); `monitor.yaml`'s `inference_table` = `deploy.yaml`'s `inference_table_prefix` + `_payload` (the monitor unpacks the table the endpoint writes); and `train.yaml`'s `model_volume_path` = the `volume_path` of the `setup.yaml` `models:` entry whose `huggingface_path` is `model_name` (setup downloads the weights, training loads them — also cross-checked by the download notebook's last cell). `train.yaml`'s `experiment_name` must stay alphanumeric/`-`/`_` (it becomes a Jobs API task key); AIR resolves it to `/Users/<user>/<experiment_name>`, and the runner notebook sets the same path for notebook runs.
- **Shard contract**: `00_setup/02_stage_training_data.py` computes `shard_id` as `pmod(xxhash64(<sft_shard_key_columns>), sft_shards)` (128 shards; both configured in `setup.yaml`) and exports the SFT records as Parquet partitioned by `shard_id` into `/Volumes/<catalog>/<schema>/<sft_volume>/<sft_table>/shard_id=N/`. `run_rank_training()` claims the directories where `N % world_size == rank` and loads them with Hugging Face `datasets` (`load_dataset("parquet", ...)`) — no Spark session on the GPU workers. Sampling is two-level: for `sample_fraction < 1` only a seeded-random subset of the rank's shard directories is loaded (hash shards are uniform, so this is equivalent to row sampling) and row-level sampling within the loaded shards lands on the exact fraction — this keeps the HF `datasets` Arrow conversion ("Generating train split") proportional to the fraction instead of always materializing the full slice, per https://docs.databricks.com/aws/en/machine-learning/ai-runtime/dataloading#load-large-delta-tables-using-volumes. Parquet was chosen because Unsloth consumes HF `datasets`, which memory-maps Parquet natively.
- **Prompt contract**: the prompt text is built once in `00_setup/02_stage_training_data.py` (Spark `concat` expressions) and the assistant response is compact JSON with keys `risk`/`action`/`reason`. Training, the serving smoke-test payload, and the load test all reuse this exact shape — the runner's sample payload and the load test both sample prompts live from the SFT table, so a changed prompt template propagates automatically once setup reruns.
- **Serving contract**: registration wraps merged HF weights in an MLflow pyfunc whose `metadata` carries `task: llm/v1/chat` and a vLLM OpenAI-server `entrypoint` listening on port 8080 (the port Model Serving expects). The entrypoint launches from the model's `artifacts/` folder, so the `--model` path must be the bare artifact name — an `artifacts/` prefix makes vLLM treat it as a Hugging Face repo id and fail with a 401. Entrypoint models must be registered with `env_pack="databricks_model_serving"` (express deployment), which needs `databricks-sdk>=0.102.0` to avoid a 5-minute timeout uploading the env-pack tarball. The pyfunc placeholder class is deliberately defined inline in the registration cell so cloudpickle serializes it by value — no `code_paths` are logged, and none are needed as long as the logged model imports nothing from the repo (the vLLM entrypoint, not the pyfunc, serves traffic). The serving container builds its environment from `deploy.yaml`'s `serving_pip_requirements` (passed as `pip_requirements` to `log_model`), not from `requirements.txt`; the default set pins `vllm==0.11.0` + `transformers>=4.56,<5` + `opencv-python-headless==4.12.0.88` (see the FIPS constraint below; transformers must be 4.x because transformers 5 removed tokenizer attributes like `all_special_tokens_extended` that this vLLM reads at startup), so the base model's architecture must be in that vLLM's supported list (the registration cell prints the architecture as a preflight). The training env runs transformers 5 while the serving env runs 4.x — the saved checkpoint and tokenizer must stay loadable by both. vLLM settings (`vllm_dtype`, `vllm_max_model_len`, `vllm_gpu_memory_utilization`) come from `02_deploy/deploy.yaml`.

### Ordering dependencies across stages

Deployment (`02_deploy/01_register_and_deploy.py`) requires a FINISHED MLflow run whose `adapter_output_dir` parameter points at saved adapters — training logs it on the rank-0 run after `trainer.save_model`, so runs that crashed before saving (and non-rank-0 runs) can never be selected. Registration/deployment is deliberately a separate stage from training so a failed registration or deployment can be rerun without re-training. Training scale is set by `notebook_gpus` in `train.yaml` for the notebook path, or the `compute` block for AIR CLI runs.

### Other constraints

- Qwen3.5 cannot currently be served via Custom LLM Serving (the reason this demo uses Qwen3): its architecture needs vLLM ≥ 0.17, but vLLM ≥ 0.15 requires `opencv-python-headless>=4.13`, whose bundled OpenSSL crashes with `FATAL FIPS SELFTEST FAILURE` on Model Serving's FIPS-enabled serverless pods (known platform issue as of June 2026). vLLM 0.11.0's opencv floor is 4.11, which is why the FIPS-safe `opencv-python-headless==4.12.0.88` pin resolves there and the serving stack works.
- The base model must be a **non-thinking** variant (hence `Qwen3-4B-Instruct-2507`, not base `Qwen3-4B`). Base Qwen3 is a hybrid reasoning model: vLLM's serving-time chat template defaults to `enable_thinking=True`, so the endpoint emits long reasoning before the JSON, truncating responses at `max_tokens` — and the training renders (which use `enable_thinking=False`) don't match the serving template, so the fine-tune can't suppress it.
- Custom LLM serving beta does not support scale-to-zero with `GPU_XLARGE` — the deploy cell raises if both are set.
- Capacity is **fixed** during beta (no autoscaling): `serving_provisioned_concurrency` sets min=max provisioned concurrency on the served entity (the optional `serving_workload_size` key is passed through as `workload_size` only when non-empty), and requests beyond provisioned capacity are rejected with instant 429s (the threshold is not user-configurable). Measured on this workload: one `GPU_MEDIUM` (A10) replica sustains ~150 successful QPS at ~1.1-1.4s latency with `max_tokens: 64-128`. Size for peak: 10K QPS needs `Large` + `GPU_XLARGE`-class capacity or multiple endpoints sharded client-side. The load generator itself caps out at (cluster task slots) × (per-worker rate) — `load_generator_workers` must not exceed available Spark task slots or achieved QPS silently falls short with zero 429s.
- The serving container has no `ninja`/`nvcc`, so FlashInfer (present in the Databricks AI base env) cannot JIT-compile kernels — vLLM crashes at sampler warmup with `FileNotFoundError: 'ninja'` unless the served entity sets `VLLM_USE_FLASHINFER_SAMPLER=0` (done via `environment_vars` in the deploy cell; this is runtime config, no re-registration needed).
- Unsloth's `use_gradient_checkpointing="unsloth"` mode is reentrant and crashes multi-GPU DDP runs under `@distributed` ("Expected to mark a variable ready only once"). The training code uses HF non-reentrant checkpointing instead (`gradient_checkpointing_kwargs={"use_reentrant": False}` in `SFTConfig`, `use_gradient_checkpointing=False` in the PEFT kwargs).
- Loading safetensors straight from `/Volumes` is minutes-slow: `from_pretrained` mmaps the weight files and every page fault becomes a small, latency-bound FUSE read. `load_unsloth_model` therefore stages any `/Volumes` model directory to node-local disk via `training_utils.stage_model_locally` before loading (top-level files only; flock-serialized so it runs once per node while other ranks wait; keyed by a size/mtime fingerprint so a retrained adapter at the same path re-stages; a staged adapter's local `adapter_config.json` is repointed at a staged copy of its volume-hosted base). Because training then loads the base from an ephemeral local path, `train.py` rewrites the saved adapter's `base_model_name_or_path` back to `MODEL_LOAD_PATH` after `trainer.save_model` — without this, registration on another machine cannot resolve the base model.
- The runner notebook derives the MLflow experiment path as `/Users/<current user>/<experiment_name>` (with `experiment_name` from `train.yaml`), matching where AIR CLI runs land — nothing user-specific is hardcoded.
- Only `README.md`, `CLAUDE.md`, `databricks.yml`, `00_setup/`, `01_train/`, `02_deploy/`, `extras/` (experimental, unsupported variants), `scripts/`, and `.github/workflows/` are version-controlled; `.claude/`, `.vscode/`, the rest of `.github/` (local agent skill copies), and other agent/IDE artifacts are gitignored.
